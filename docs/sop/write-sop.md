# How to Write an SOP for CommunityFix

## What is an SOP?

An SOP (Standard Operating Procedure) is a step-by-step guide that helps community members perform a recurring task consistently. On CommunityFix, SOPs ensure that everyone — regardless of experience — can contribute effectively by following the same process.

## When to write an SOP

Write an SOP when you notice a task that:

- Is repeated by multiple people.
- Leads to inconsistent results when done without guidance.
- Requires decisions that can be standardized into clear steps.

## Structure

Every SOP follows this format:

### Title

Use the pattern: **"How to [action]"**

The title should describe the task, not the outcome. Keep it short and scannable.

> **Good:** "How to Structure an Issue"
> **Bad:** "Issue Structuring Best Practices and Guidelines for New Users"

### Steps

Break the task into numbered steps. Each step should:

- Start with a **verb** (Identify, Ask, Write, Attach).
- Answer **one question** or complete **one action**.
- Be short enough to hold in your head while doing it.

Use bold for the key question or action at the top of each step, then explain underneath.

> **Example:**
>
> ## Step 1: Identify the core problem
>
> Ask yourself: **if someone on the other side of the world had this same problem, what would they call it?**
>
> Strip away your location, your specific causes, your local context. What remains is your issue title.

### Examples

Use blockquotes (`>`) for examples throughout the steps. Good examples are:

- **Concrete** — use real or realistic scenarios, not abstract placeholders.
- **Universal** — pick examples most people will recognize (Amazon, not a niche local case).
- **Paired** — when possible, show a good and bad example side by side.

### Checklist

End with a quick checklist that summarizes the key decisions. Use markdown checkboxes so people can mentally tick them off.

```markdown
- [ ] First check
- [ ] Second check
```

## Writing style

- **Use "you" and direct questions.** SOPs are instructions, not essays. Talk to the reader.
- **Keep paragraphs short.** One to three sentences max. If a paragraph is getting long, it probably needs to be split into a sub-step.
- **Bold the key phrase in each step.** People scan before they read. The bolded part should be enough to follow the SOP on a second pass.
- **Avoid jargon.** If a term isn't obvious, explain it the first time you use it. Remember the community is global and multilingual.
- **Use blockquotes for examples, not for instructions.** Instructions are the main text. Examples illustrate them.

## Markdown formatting rules

- Use `#` for the SOP title, `##` for steps, `###` for sub-sections within steps.
- Use `>` blockquotes for examples.
- Use `- [ ]` checkboxes for the final checklist.
- Use **bold** for key phrases and actions, not for entire sentences.
- Use `---` horizontal rules only to separate the main content from the checklist at the end.
- Do not use tables unless comparing structured data. Prefer prose.
- Do not use images. SOPs should be readable as plain text.

## Quick checklist

- [ ] Does the title follow the "How to [action]" pattern?
- [ ] Does each step start with a verb and answer one question?
- [ ] Are examples concrete, universal, and in blockquotes?
- [ ] Does it end with a scannable checklist?
- [ ] Is the language direct, jargon-free, and addressed to "you"?
- [ ] Could someone with no prior context follow it on their first read?
